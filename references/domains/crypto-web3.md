# Domain Overlay: Crypto / Web3

Apply these Five Truths calibrations when analyzing crypto/web3 decisions.

| Truth | Crypto/Web3 Calibration |
|-------|------------------------|
| **User Truth** | Crypto-native users are sophisticated and adversarial — they will find and exploit edge cases, front-run mechanisms, and game incentive structures. Design assuming adversarial users, not cooperative ones. |
| **Strategic Truth** | Protocol + application layer separation determines long-term value capture. The question isn't "should we build a protocol or an app?" — it's "which layer captures value in our specific market structure?" Most value has historically accrued at the protocol layer, but most teams can't build at that layer. |
| **Economic Truth** | Token economics must align incentives across all participant types from day one — users, validators, developers, speculators. Misaligned tokenomics can't be patched post-launch. If the only reason to hold the token is speculation, the economics will eventually collapse. |
| **Macro-Political** | Regulatory clarity varies dramatically by jurisdiction. SEC vs CFTC jurisdiction is unresolved in the US; MiCA is live in the EU; Asia varies by country. Structure for the most restrictive likely outcome, not the most permissive current state. |
| **Execution Truth** | Smart contract bugs are irreversible and catastrophic. Audit cadence must match shipping cadence — not a one-time pre-launch exercise. Post-launch, every upgrade is a governance and security event. |

## Web3-Specific Kill Criteria

- If token design creates mercenary liquidity that exits at first sign of trouble → tokenomics are broken
- If the protocol has no credible path to decentralization → regulatory exposure as a centralized intermediary
- If the core use case doesn't require trustlessness → why is this on a blockchain? The answer must be clear.
- If audit hasn't been completed on any contract holding user funds → shipping is the wrong decision
